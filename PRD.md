## 一、项目概述

### 1.1 项目背景

微动（WEID）是一家医美/美容行业的连锁门店运营公司，拥有覆盖全国多城市的线下门店网络。公司业务涵盖美容服务预约、客户关系管理、前端销售团队管理、门店运营等多个环节。当前需要一个统一的内部后台管理系统，将分散在各渠道的业务数据进行集中化管理，提升运营效率。

### 1.2 项目目标

1. **统一管理入口**：将门店运营、客户管理、订单处理、财务核算等核心业务统一到一个后台系统中
2. **数据可视化**：提供多维度业务数据看板，辅助管理层决策
3. **流程标准化**：规范预约管理、审批流程、财务核算等业务流程
4. **前端绩效管理**：支持前端销售团队的床位分配、业绩追踪、加微转化等管理需求

### 1.3 技术约束

- **技术栈**：纯 HTML / CSS / JavaScript，无外部框架依赖（如 React、Vue 等）
- **图标库**：Remix Icon v3.5.0（CDN 引入）
- **数据层**：当前为静态原型阶段，使用模拟数据；后续可对接后端 API
- **浏览器兼容**：主要面向 Chrome / Edge 等现代浏览器
- **设计规范**：已有完整的设计系统文档（design-system.md），新页面需严格遵循

### 1.4 系统定位

## 二、用户角色与使用场景

### 2.1 用户角色定义

| 角色 | 职责描述 | 核心使用模块 | 使用频率 |
|------|---------|-------------|---------|
| **运营管理员** | 管理门店信息、产品上架、预约配置 | 门店列表、产品管理、首页 | 每天 |
| **前端销售** | 管理客户关系、跟踪订单、查看业绩数据 | 客户、订单、数据 | 每天 |
| **数据分析师** | 分析业务数据、生成报表、监控关键指标 | 数据、核算 | 每天/每周 |
| **财务人员** | 管理收支记录、财务核算、对账 | 核算、收支记录 | 每周/每月 |
| **审批管理员** | 处理各类审批申请 | 审批 | 每天 |
| **推广人员** | 管理推广咨询、客户引流 | 推广咨询 | 每天 |
| **高管/老板** | 查看全局数据、审批重要事项 | 首页、数据、审批 | 偶尔 |

### 2.2 典型使用场景

#### 场景1：门店预约项目管理（运营管理员）
> 运营管理员需要根据业务需求，配置各门店的预约项目和开关状态。例如：上海徐汇店开启了 S10、K10-K30 等项目的预约功能，需要批量修改多个门店的预约配置。

#### 场景2：前端床位管理（前端销售/管理者）
> 管理者需要按月为前端销售员工分配床位额度（默认每月30个），跟踪每个人的床位使用情况、爽约率和到店转化数据。

#### 场景3：业务数据看板（数据分析师/高管）
> 需要查看前端预约、前端业绩、加微转化留存等核心业务数据，支持按大区、部门、小组、人员等多维度筛选和分析。

## 四、非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面首次加载 | ≤ 2秒（本地静态文件） |
| 表格渲染 | 支持 100+ 行数据流畅滚动 |
| 筛选响应 | 筛选操作后 ≤ 300ms 更新视图 |
| 内存占用 | 单页面内存 ≤ 50MB |

### 4.2 兼容性要求

| 浏览器 | 最低版本 |
|--------|---------|
| Chrome | 80+ |
| Edge | 80+ |
| Firefox | 78+ |
| Safari | 13+ |

### 4.3 设计规范

- 严格遵循 `design-system.md` 中定义的色彩系统、字体系统、间距系统
- 新页面必须基于 `master-template.html` 母版创建
- 组件样式保持一致性：按钮、表格、筛选标签、状态标签等
- 圆角默认 2px，遵循设计系统变量

### 4.4 代码规范

- 每个页面为独立 HTML 文件，内联 CSS 和 JavaScript
- CSS 变量统一使用 `:root` 定义
- JavaScript 使用原生 ES5/ES6 语法，不依赖外部库
- 代码注释使用中文

### 4.5 可维护性

- 页面结构统一：顶部导航 → 侧边菜单 → 内容区（面包屑 → 标签页 → 筛选 → 表格）
- 新增页面可快速复制母版模板
- 菜单激活状态通过 CSS class 控制，易于维护

## 六、验收标准

### 6.1 功能验收

1. 所有 P1 功能完整实现，交互流程正常
2. 菜单导航正常跳转，面包屑路径正确
3. 表格数据展示完整，筛选功能正常
4. 开关、按钮等交互组件响应正确
5. 已完成的两个页面（门店预约项目、前端床位）作为质量基准

### 6.2 设计验收

1. 所有页面视觉风格与已有页面一致
2. 颜色、字体、间距、圆角等符合设计规范
3. 交互状态（hover、active、focus）完整
4. 暗色主题（侧边栏、顶栏）与其他页面一致

### 6.3 技术验收

1. 纯 HTML/CSS/JS 实现，无框架依赖
2. 页面加载无报错
3. 浏览器兼容性测试通过
4. 代码结构清晰，可维护

## 七、风险与约束

| 风险项 | 影响 | 应对策略 |
|--------|------|---------|
| 纯前端无后端 | 数据无法持久化，无法真正 CRUD | 当前阶段为 UI 原型展示，使用模拟数据 |
| 业务逻辑不明确 | 部分页面功能定义模糊 | 优先完成已有明确参照的页面，其余与业务方确认后再开发 |
| 单文件页面膨胀 | 复杂页面 HTML 文件过大 | 保持代码结构清晰，CSS/JS 分区注释 |
| 菜单结构可能变化 | 已有两个参考页面的菜单不完全一致 | 以 master-template.html 的菜单为准，保持统一 |
